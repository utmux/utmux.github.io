本节主要只谈元素（节点）的插入和搜寻。RB-tree 提供两种插入操作：insert_unique() 和 insert_equal()，前者表示被插入节点的键值（key）在整棵树中必须独一无二（因此，如果树中已存在相同的键值，插入操作就不会真正进行），后者表示被插入节点的键值在整棵树中可以重复，因此，无论如何插入都会成功（除非空间不足导致配置失败）。这两个函数都有数个版本，以下以最简单的版本（单一参数，用以表示将被插入的节点实值（value））作为说明对象。注意，虽然只指定实值，但 RB-tree 一开始即要求用户必须明确设定所谓的 KeyOfValue 仿函数，因此，从实值（value）中取出键值（key）是毫无问题的。

5.3 set

set 和 map 不一样，不同时拥有 value 和 key，set 元素的键值就是实值。且 set 不允许两个元素拥有相同的值，set 会自动排序，且 set 的迭代器是 const iterator 即，禁止改写。和 list 一样，插入或者删除元素不会导致已有的迭代器指向的元素错位或者失效。已有的元素错位或者失效的原因（比如 std::vector 和 std::dequeue）是由于动态空间分配，导致元素搬移到新的地方去，从而使得原有的迭代器仍然指向旧地址。

除了 set 还有一个 multiset，这俩个都提供了：取 交集，并集，差集 和 对称差集 这几种运算。

5.4 map

map 中不允许修改键，这是因为其红黑树排序的依据就是键，如果修改会导致排序混乱。map 可以修改键对应的值。

map 和 set 的不同之处是，其比较函数，比较的是键。除此之外 map 的迭代器是允许修改的，其可以修改 key 对应的 value，以及 map 中实现了 operator[] 函数，使得其能够通过键修改值，或者读取到键所对应的值。该实现是通过 _insert() 函数插入一个空的 value 实现的，由于 map 和 set 中不允许重复键值，如果遇到重复键值，则会返回对应的键。即使不存在重复对，由于是空的 value 所以并不会插入成功。

5.5 multimap

和上面的 map 不同之处是插入函数调用 insert_equal() 函数，而不是 insert_unique() 函数。其他和 map 没有什么多大的区别。

5.6 hashtable

前面的算法课中有一个非常简易的 hash function 映射函数如下：

为了避免碰撞，N 是一个特殊的值。下面我们来看一下 hashtable 为了处理碰撞所采用的几种方式。

  1. 线性探测：
    在表格足够大，负载系数足够小时，此时只管从头到尾，循环转一圈，哪里有空放哪里。平均情况则是每次插入需要访问一般的表格才能找到空位插入。不过其存在主集团现象，如下图，除非 hash function 映射的出来的结果恰好在 4-7 之间，否则其他函数按照从左往右探测，必定会优先使用 3 号，由于冲突检测这浪费了大量的时间，而且随着 3，4 的使用，情况会越来越恶化。

  2. 二次探测
    二次探测是为了解决一次探测导致的问题，其探测方式不再是从左到右一个一个探测，而且使用二次函数探测，分别探测：n+1,n+4,n+9,n+16 的位置。二次探测带来了更多的问题，比如其碰撞概率，以及在有空位的情况下，按照这种移动方式，能否在任意一个位置开始可以访问到任意其他位置。如何判断 hashtable 此时容量已满？如何获取此时负载系数来判断需要动态增长。根据下面：

    n2−(n−1)2=2n−1n^2-(n-1)^2 = 2n-1 n2−(n−1)2=2n−1

    每次步长为上次步长右移一位再减去 1。

    且根据证明，当 N 为质数，且负载小于 0.5 时，此时最多不超过两次即可成功插入。

    二次探测带来了次集团问题，即当负载比较大时，会导致两个元素得到的位置冲突。

5.6.5 hash_set 和 hash_map

这里和前面的红黑树实现的 set 和 map 进行区分，由于红黑树中插入是有序插入进行了排队。而再 hashtable 实现的 set 和 map 中，这两个是没有排序的。hashtable 决定了该种数据结构只适合快速查找，而不适合排序。
