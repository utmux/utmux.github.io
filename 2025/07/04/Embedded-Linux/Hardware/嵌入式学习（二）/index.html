2. 设备创建流程

sequenceDiagram
    participant Main as 主程序
    participant TypeSys as 类型系统
    participant Class as 设备类
    participant Instance as 设备实例
    
    Main->>TypeSys: type_register_static(&ask100fb_info)
    activate TypeSys
    TypeSys->>Class: ask100fb_class_init() 执行一次
    deactivate TypeSys
    
    Main->>TypeSys: qdev_create(NULL, TYPE_100ASKFB)
    activate TypeSys
    TypeSys->>Instance: object_new(TYPE_100ASKFB)
    activate Instance
    Instance->>Instance: 分配内存(sizeof(ASK100FbState))
    Instance->>Instance: ask100fb_init() 实例初始化
    deactivate Instance
    TypeSys->>Instance: qdev_init_nofail(dev)
    activate Instance
    Instance->>Instance: device_realize()
    Instance->>Instance: ask100fb_realize() 设备实现
    deactivate Instance
    deactivate TypeSys

3. 具体执行顺序

  1. 类型注册阶段（在 QEMU 启动时执行）：

    • 调用 type_register_static(&ask100fb_info)
    • 触发 类初始化 ask100fb_class_init()
      • 这个函数 只执行一次，无论创建多少设备实例
      • 设置设备类别和 realize 回调

  2. 设备实例化阶段：

    • 调用 qdev_create(NULL, TYPE_100ASKFB)
    • 触发以下顺序：
      a. 实例初始化 ask100fb_init()
      • 为每个设备实例单独执行
      • 初始化设备特定状态
      • 设置 I/O 内存区域
        b. 设备实现 ask100fb_realize()
      • 通过类初始化中设置的 realize 回调触发
      • 执行设备的具体实现逻辑

4. 关键区别：类初始化 vs 实例初始化

调用时机 类型注册时 每次创建设备实例时

主要职责 设置类级别的行为 初始化实例特定状态
参数 ObjectClass *klass Object *obj
典型操作 设置 realize 回调

特别注意事项

  1. 类初始化的唯一性：

    • 即使创建多个 LCD 设备实例，ask100fb_class_init() 也只会执行一次
    • 所有实例共享相同的类结构和 realize 回调

  2. 实例初始化的独立性：

    • 每个设备实例都有自己的 ask100fb_init() 调用
    • 每个实例维护独立的状态（ASK100FbState）

  3. Realize 的延迟执行：

    • realize 在设备被添加到总线时触发
    • 可以在实例初始化之后设置设备属性
    • 允许依赖其他设备完成初始化

2. sysbus_realize_and_unref() 中的 unref 含义

✅ 功能

  • 调用 .realize 回调函数（用于设备逻辑初始化）
  • 然后自动调用 object_unref() 减少一次引用计数，防止内存泄漏

📌 使用场景

当你通过 qdev_new() 创建设备实例时，QEMU 会自动增加一次引用计数。

❗ 不使用 unref 的后果

如果不调用 unref，会导致设备对象无法释放，造成内存泄漏。

⚠️ 注意事项

  • 如果设备是嵌入式结构体成员（非动态分配），不需要调用 unref
  • 推荐使用 sysbus_realize_and_unref()，它是标准做法

3. MMIO 注册与映射机制

✅ 主要函数

函数 作用
memory_region_init_io() 初始化一个 I/O 内存区域（如寄存器区域）
sysbus_init_mmio() 告诉系统总线该设备有一个 MMIO 区域
sysbus_mmio_map() 将 MMIO 区域映射到指定物理地址
memory_region_add_subregion() 底层函数，完成实际的地址映射

4. 其他知识点

1. 使用 BusyBox 制作轻量级 initramfs 环境

在嵌入式开发中，为了快速验证内核驱动功能，可以使用静态编译的 BusyBox 构建一个最小化的 Linux 用户空间环境。

🔧 主要步骤包括

  • 静态编译 BusyBox

    • 配置 BusyBox 为静态链接模式（Build Options → Build static binary）
    • 编译后得到一个不依赖动态库的 busybox 可执行文件

  • 构建 rootfs 目录结构

  • 创建设备节点（mknod）

  • 编写 init 脚本

    注意：init 脚本需要有执行权限，并且第一行为 shebang（#!/bin/sh）

  • 打包成 initramfs

  • QEMU 启动命令示例

✅ 优势

  • 快速启动，无需完整根文件系统
  • 适合调试内核模块、驱动加载顺序等
  • 易于定制和修改，方便测试

2. 使用 GDB + QEMU 调试 Linux 内核

通过 QEMU 搭配 GDB，可以在用户态对 Linux 内核进行源码级调试。

🛠️ 准备工作

  • 内核配置开启调试信息

    • CONFIG_DEBUG_INFO=y
    • CONFIG_KGDB=y（可选）
    • CONFIG_MODULES=y（如需加载模块）

  • QEMU 启动参数添加调试支持

    • -s：启用 gdbserver，默认监听端口 1234
    • -S：暂停 CPU，等待 GDB 连接后才开始运行

💡 使用 GDB 连接调试

  • vmlinux 是包含调试符号的内核镜像
  • 可以设置断点、查看寄存器、单步执行等

🧪 典型调试场景

  • 在 start_kernel 设置断点
  • 查看驱动加载过程中的指针访问异常
  • 调试中断处理流程
  • 分析 oops 或 panic 崩溃现场

3. LCD 控制器模型与设备树理解

在学习 LCD 控制器驱动开发过程中，进一步理解了其硬件抽象方式及设备树配置原理。

🖥️ LCD 控制器模型核心要点

  • 控制器寄存器存储的是显存的总线地址

    • 例如：帧缓冲起始地址寄存器（FB_BASE_REG）保存的是物理内存地址
    • 控制器通过 DMA 方式直接从该地址读取显存数据并输出到屏幕

  • 显存需求随分辨率增大而增加

    • 800x480 RGB888：约 1.2MB
    • 1920x1080 RGB888：约 6MB

  • 显存映射机制

    • 在驱动中通过 dma_alloc_coherent() 或 ioremap() 分配显存
    • 将物理地址写入控制器寄存器供硬件访问

📄 设备树配置详解

  • LCD 设备节点示例

  • 参数含义参考数据手册时序图

    • clock-frequency: 像素时钟频率（Hz）
    • hactive/vactive: 屏幕有效像素宽高
    • hfront-porch, hsync-len, hback-porch: 水平同步信号时序参数
    • 类似地，垂直方向也有相应参数

  • 设备树与驱动匹配机制

    • 驱动通过 .of_match_table 匹配设备树节点的 compatible 字段
    • 匹配成功后触发 probe() 函数初始化控制器

✅ 开发建议

  • 对照 LCD 数据手册填写设备树参数
  • 使用 fb_set_var() 更新显示参数
  • 使用 framebuffer_alloc() 创建 fb_info 结构体
  • 避免频繁申请释放显存，使用一致性内存（DMA 内存）

🧱 配置流程如下

✅ 1. 使用 NXP Pin Tool 配置引脚功能
  • 打开 NXP Pin Tool
  • 选择芯片型号（如 MCIMX6Y2）
  • 设置以下选项：
    • MUX Mode：选择复用功能（如 UART1_RX_DATA、LCD_CLK 等）
    • Pull-up/down：选择是否启用上下拉电阻
    • Drive Strength：设置驱动能力（低、中、高）
    • Slew Rate：斜率控制（快/慢）
    • Open Drain：是否启用开漏模式
